# CineVault — Дизайн-документ (Финальная версия)

**Версия:** 2.0
**Тип:** Настольное веб-приложение (SPA, локальный запуск через лаунчер)
**Назначение:** Персональный менеджер коллекции фильмов с системой случайного выбора («Колесо Судьбы»)

> Документ объединяет исходную спецификацию CineVault с дополнениями из уточнённого технического задания заказчика. В случаях расхождений (архитектура запуска, механизм колеса, цветовая палитра) приоритет отдан реализованному решению CineVault — с пояснением причины расхождения в соответствующем разделе.

---

## 1. Концепция

CineVault — персональная система управления собственной библиотекой фильмов. Приложение предназначено для:

- хранения фильмов и их метаданных;
- объединения фильмов во франшизы;
- хранения пользовательских оценок от нескольких зрителей;
- ведения истории просмотров;
- организации фильмов по категориям;
- случайного выбора фильма или франшизы через механику колеса с выбыванием.

Авторизация отсутствует. Все данные принадлежат одному пользователю, хранятся локально и никуда не передаются.

### 1.1 Ключевые принципы
- **Локальность.** Все данные — в `localStorage`, сервер не нужен, интернет не требуется (кроме загрузки постеров по внешнему URL, если используются).
- **Самодостаточность.** Один HTML-файл с инлайн CSS/JS — легко переносить, не требует сборки.
- **Мгновенный запуск.** Через `launch.py` открывается как отдельное локальное приложение одним двойным кликом, без установки зависимостей.
- **Модульность бизнес-логики.** Логика данных отделена от отрисовки интерфейса, что позволяет менять UI или слой хранения, не переписывая правила приложения.

## 2. Архитектура

### 2.1 Технологический стек
| Слой | Технология |
|---|---|
| Разметка/логика | Vanilla HTML5 + JS (ES6+), без фреймворков |
| Стили | Чистый CSS3 с переменными (`:root`), без препроцессоров |
| Хранилище | `localStorage` (JSON-сериализация), с архитектурой, допускающей замену на IndexedDB без переработки бизнес-логики |
| Шрифты | Google Fonts: `Bebas Neue` (заголовки), `DM Sans` (текст) |
| Запуск | Python 3 (`http.server` + `webbrowser`) — оборачивает `index.html` в локальный веб-сервер и открывает системный браузер |

### 2.2 Программные модули
Приложение логически разделено на независимые зоны ответственности (реализованы как объекты/секции внутри единого JS-файла, с возможностью вынесения в отдельные модули при переходе на сборку):

- **Movie** — CRUD фильмов, вычисление среднего рейтинга;
- **Franchise** — CRUD франшиз, агрегация рейтинга по входящим фильмам;
- **Category** — CRUD категорий, включая вложенность;
- **Rating** — добавление/пересчёт оценок;
- **Watch** — статус и дата просмотра, ручные и автоматические переходы;
- **Random Session (Wheel)** — независимая сессия розыгрыша с пулом участников и историей выбывания;
- **Storage** — сериализация в `localStorage`, точка расширения под IndexedDB;
- **Statistics** — агрегированные показатели библиотеки;
- **Search/Sort** — фильтрация и упорядочивание каталога;
- **UI / Navigation** — отрисовка экранов и переключение между ними;
- **Settings** — управление сейвами, категориями, служебными параметрами.

Каждый модуль владеет только своей областью данных; UI-слой не содержит бизнes-правил напрямую, а вызывает методы модулей.

### 2.3 Почему не Electron
Реализация через Electron была рассмотрена первой (для честного `.exe`), но недоступна в текущей рабочей среде из-за отсутствия сетевого доступа к npm-реестру для установки `electron`/`electron-builder`. Взамен выбран паттерн **«локальный сервер + системный браузер»**: Python-скрипт поднимает `http.server` на `localhost:7432` и автоматически открывает вкладку браузера — работает как отдельное системное приложение.

**Путь к настоящему `.exe`:** `src/index.html` можно напрямую упаковать в Electron (`electron-builder --win`) или обернуть `launch.py` через PyInstaller на машине с доступом в интернет — оба варианта дадут двойной клик → готовое окно приложения без изменений в коде.

### 2.4 Структура файлов
```
moviemanager/
├── src/
│   └── index.html      # Всё приложение: HTML + CSS + JS в одном файле
├── launch.py            # Лаунчер: поднимает сервер и открывает браузер
└── package.json          # Заготовка под будущую Electron-сборку
```

## 3. Структура данных

### 3.1 Movie
```js
{
  id: string (uuid),
  title: string,
  cover: string,               // URL постера, может быть пустым
  duration: number | null,      // минуты (расширение)
  description: string | null,   // описание (расширение)
  releaseYear: number | null,   // год выпуска (расширение)
  catId: string | string[],     // категория(и) — поддержка множественной принадлежности
  watched: boolean,
  inRoll: boolean,               // участвует ли в текущем ролле колеса
  ratings: [{ u: string, value: number, date: ISO }],  // автор, оценка 1–10, дата
  addedAt: ISO-дата,
  watchedAt: ISO-дата | null     // может быть изменена пользователем вручную
}
```

### 3.2 Franchise
```js
{
  id: string (uuid),
  name: string,
  poster: string | null,
  movieIds: string[],
  inRoll: boolean,
  watched: boolean
}
```
Франшиза — самостоятельная сущность, не являющаяся фильмом; объединяет несколько фильмов и участвует в колесе как единый элемент независимо от числа фильмов внутри.

### 3.3 Category
```js
{
  id: string,
  name: string,
  parentCategoryId: string | null   // поддержка неограниченной вложенности (расширение)
}
```

### 3.4 Random Session (сессия колеса)
```js
{
  id: string,
  createdAt: ISO,
  entityPool: (Movie|Franchise)[],   // текущий пул участников
  removedEntities: (Movie|Franchise)[],  // лог выбывших
  winner: Movie|Franchise|null
}
```
Сессия полностью изолирована: выбывание сущности удаляет её **только из текущей сессии**, основной каталог не изменяется. После определения победителя сессия удаляется; сессии не пересекаются друг с другом.

### 3.5 Saves (сейвы)
```js
{ "Игрок 1": 3, "Игрок 2": 3 }
```

При первом запуске, если база фильмов пуста, генерируется демонстрационный набор — чтобы интерфейс не встречал пользователя пустым экраном.

## 4. Карта экранов (Views)

### 4.1 Каталог
Показывает непросмотренные фильмы и франшизы.

Карточка фильма отображает: постер, название, год выхода, длительность, категории, средний рейтинг, количество оценок, статус и дату просмотра (если применимо), принадлежность франшизе.

Поддерживаются: создание, редактирование, удаление, поиск, фильтрация, сортировка (см. разделы 4.6–4.7), пакетное включение в ролл.

Карточка франшизы: список входящих фильмов, средний рейтинг по всем оценённым фильмам, кнопки «В ролл» / «Добавить фильм» / «Удалить».

### 4.2 Категории
Категории организованы в виде **дерева с неограниченной вложенностью**; фильм может принадлежать нескольким категориям одновременно.

Поддерживаются: создание, редактирование, удаление, вложенные подкатегории, фильтрация по конкретной категории, по нескольким категориям сразу, по родительской категории (с учётом всех дочерних).

Внутри категории — пакетный ввод в ролл: «В ролл первые: N» + кнопка «Применить».

### 4.3 Просмотренные
Отображает просмотренные фильмы и полностью просмотренные франшизы (франшиза — раскрываемая группа со списком входящих фильмов).

**Способы попадания в просмотренные:**
1. **Ручное добавление** — пользователь в любой момент отмечает фильм просмотренным; автоматически устанавливается статус и текущая дата, либо пользователь сразу указывает произвольную дату просмотра.
2. **Автоматическое добавление** — после победы в колесе. Если победила франшиза — просмотренными становятся **все фильмы внутри неё одновременно**, всем присваивается одна и та же дата просмотра, франшиза считается просмотренной полностью.

**Изменение статуса:** пользователь может снять отметку о просмотре, повторно отметить фильм просмотренным, изменить дату просмотра в любой момент — все изменения немедленно сохраняются и обновляют статистику.

Поддерживается сортировка: по названию, по рейтингу, по дате просмотра.

### 4.4 Колесо Судьбы

**Механика (сохранена и расширена относительно оригинала):**
1. Перед запуском создаётся отдельная **Random Session**; в пул попадают фильмы и франшизы с флагом `inRoll = true`, не просмотренные;
2. Франшиза для колеса — один элемент, независимо от количества фильмов внутри;
3. Требуется минимум 2 участника;
4. Колесо — `<canvas>`, равные сегменты, чередующиеся цвета, подписи по радиусу, стрелка фиксирована слева;
5. Вращение — анимация 4 секунды с замедлением (`ease-out-quart`), 8+ оборотов + случайный довесок;
6. Сегмент под стрелкой — **выбывающий** участник (механика «на вылет», не «победитель за один спин»); выбывание удаляет сущность только из текущей сессии, основной каталог не меняется;
7. Пользователь подтверждает выбывание либо перекручивает колесо повторно;
8. Процесс продолжается, пока не останется одна сущность — она объявляется победителем;
9. Если победитель — фильм: он автоматически отмечается просмотренным с текущей датой. Если победитель — франшиза: **все входящие в неё фильмы** автоматически отмечаются просмотренными с одной датой, франшиза — просмотренной полностью;
10. После определения победителя сессия удаляется; полноэкранная целебрация с конфетти и автопереход в «Просмотренные».

**О сервисе Wheel of Names.** Согласно исходному ТЗ, колесо должно использовать Wheel of Names как механизм случайного выбора. Проверка их официальной документации показала: **публичного API для программной передачи списка участников и получения результата обратно в стороннее приложение у сервиса нет** — сами разработчики подтверждают, что embedding недоступен, а единственная официальная альтернатива — статичная iframe-ссылка на предварительно созданное на их сайте колесо в режиме «только вращение», без текстового поля и кнопок. Это не позволяет:
- динамически передавать список фильмов из каталога CineVault;
- получать результат обратно в приложение для отметки «просмотрено»;
- реализовать сессионную изоляцию и историю выбывания, требуемые ТЗ.

**Решение (приоритет отдан этому варианту):** собственное колесо на `<canvas>`, визуально и по ощущениям приближенное к Wheel of Names, но полностью интегрированное с базой данных CineVault по описанной выше механике.

### 4.5 Статистика
Отображаются агрегированные показатели, пересчитываемые автоматически при любом изменении данных:

- общее количество фильмов;
- количество просмотренных / непросмотренных фильмов;
- количество франшиз и количество просмотренных франшиз;
- средняя оценка библиотеки;
- общее количество оценок;
- суммарная длительность всех фильмов и отдельно — просмотренных;
- самая высокая и самая низкая оценка в библиотеке;
- самая длинная франшиза (по числу фильмов).

### 4.6 Поиск
Выполняется по: названию, описанию, году выпуска, категории, франшизе. Поддерживается частичное совпадение (живой поиск без подтверждения).

### 4.7 Сортировка
Каталог поддерживает сортировку по: названию, году, длительности, средней оценке, количеству оценок, дате просмотра.

### 4.8 Настройки
Раздел управления сервисными параметрами: сейвы и игроки (создание/редактирование/удаление), управление категориями верхнего уровня, точка входа для импорта/экспорта библиотеки (см. раздел 8).

## 6. Дизайн-система

### 6.1 Направление
**Кинематографичный тёмный минимализм** — атмосфера кинотеатрального фойе поздним вечером: глубокий почти-чёрный фон, золотая подсветка как в неоновых вывесках старых кинотеатров, крупная плакатная типографика для заголовков.

> В уточнённом ТЗ предлагалась альтернативная палитра (фон `#262626`, карточки `#3A3A3A`, акцент `#10B981`, текст `#ECECEC`) в более нейтральном сером минимализме. Приоритет отдан кинематографической золотой палитре CineVault как более выразительной и тематически точной для приложения о кино; требования к стилю («минимализм, карточное отображение, плавные анимации, отсутствие визуального шума, единый современный стиль») выполнены в рамках выбранной палитры.

### 6.2 Палитра (действующая)
| Токен | Значение | Назначение |
|---|---|---|
| `--ink` | `#0a0b0f` | Базовый фон приложения |
| `--surface` | `#10131a` | Сайдбар, шапка |
| `--card` | `#161c27` | Карточки, панели |
| `--card2` | `#1d2535` | Вложенные элементы, hover-состояния |
| `--gold` | `#e8b84b` | Основной акцент |
| `--gold2` | `#f5d278` | Светлый акцент (hover) |
| `--red` / `--red2` | `#e85050` / `#ff6b6b` | Удаление, выбывание в колесе |
| `--text` / `--muted2` / `--muted` | `#e8eaf0` / `#9299aa` / `#6b7385` | Иерархия текста |

### 6.3 Типографика
- **Bebas Neue** — заголовки, лого, статус колеса, кнопка «КРУТИТЬ»;
- **DM Sans** — весь интерфейсный текст.

### 6.4 Motion
Fade+slide при появлении карточек; подъём карточки и увеличение постера при hover; вращающееся кольцо-свечение вокруг колеса; пружинящий трофей на экране победителя; выезжающие тосты; комбинация fade+scale+translateY для модалок.

### 6.5 Компоненты
Единый набор: кнопки (`primary`/`ghost`/`danger`/`roll`), карточки фильма и франшизы, аккордеон категорий (с поддержкой вложенности), строки истории, панели колеса, модалки с единым API, тосты, пустые состояния — все на CSS-переменных для лёгкой смены темы.

### 6.6 Разделы приложения (навигация)
Главная · Каталог · Категории · Франшизы · Просмотренные · Колесо · Статистика · Настройки.

## 7. Оценки

Каждый фильм поддерживает неограниченное количество оценок. Каждая оценка содержит автора, числовую оценку (1–10) и дату добавления. Средняя оценка фильма вычисляется автоматически и пересчитывается при любом изменении или удалении оценки. Средняя оценка франшизы — среднее значение средних оценок всех входящих в неё фильмов.

---

## 8. Импорт и экспорт

Поддерживается экспорт всей библиотеки в единый JSON-файл. Импорт полностью восстанавливает: фильмы, категории (включая иерархию), франшизы, оценки, историю просмотров. Это резервный механизм переноса данных между устройствами/браузерами, поскольку `localStorage` не синхронизируется автоматически.

---

## 9. Хранение данных

Все изменения сохраняются автоматически, без отдельного действия «Сохранить». Текущий слой хранения — `localStorage`; архитектура (см. раздел 2.2, модуль Storage) спроектирована так, чтобы в дальнейшем заменить его на IndexedDB без переработки бизнес-логики остальных модулей.

## 11. Новое, объединяющее обе спецификации

- Поиск по названию, описанию, году, категории, франшизе; сортировка по 6 критериям;
- Расширенные поля фильма: длительность, описание, год выпуска;
- Вложенные категории и множественная принадлежность фильма категориям;
- Полноценный раздел «Статистика» с 11 агрегированными показателями;
- Ручное управление датой просмотра, возможность снять/вернуть отметку;
- Импорт/экспорт библиотеки в JSON;
- Раздел «Настройки»;
- Живая статистика в сайдбаре, тост-уведомления, полноэкранная целебрация победителя с конфетти;
- Возможность перекрутить колесо перед подтверждением выбывания;
- Пустые состояния с осмысленными подсказками;
- Лаунчер для запуска как отдельного «приложения» одним кликом.

---

## 12. Общие требования

- Все сущности имеют уникальные идентификаторы (`uuid`).
- Все вычисляемые значения (средние оценки, статистика) пересчитываются автоматически при любом изменении.
- Все изменения сохраняются автоматически, без явного действия пользователя.
- Вся бизнес-логика реализована на стороне JavaScript, независимо от UI-слоя.
- Пользовательские данные не передаются на сервер.
- Случайные сессии колеса полностью изолированы друг от друга и от основного каталога.
- Архитектура поддерживает дальнейшее расширение функциональности без переработки существующих модулей.

## 13. Известные ограничения и рекомендации на будущее

1. **Данные привязаны к браузеру.** `localStorage` не синхронизируется между устройствами — компенсируется механизмом импорта/экспорта (раздел 8), но перенос остаётся ручным действием, а не автосинхронизацией.
2. **Постеры по внешним URL** требуют интернет-соединения при отображении — рекомендация: локальное хранение изображений (base64 или файловая система при Electron-сборке).
3. **Настоящий `.exe`** не собран в этой среде из-за отсутствия сетевого доступа к npm. Проект готов к сборке через Electron Builder или PyInstaller на машине с интернетом без изменений в коде.
4. **Сейвы** реализованы как учётный механизм расхода счётчика, а не автоматическая защита конкретного участника от выбывания в момент розыгрыша — соответствует исходному поведению, но может быть расширено до полноценной игровой механики.
5. **Переход на IndexedDB** предусмотрен архитектурно, но не реализован — рекомендуется при превышении объёма данных, комфортного для `localStorage` (постеры в base64 могут быстро исчерпать лимит ~5–10 МБ).

## 14. Итог

CineVault объединяет исходную реализацию и уточнённые требования заказчика в единую непротиворечивую спецификацию: расширенная модель данных (длительность, описание, год, вложенные категории, даты оценок), полноценные поиск/сортировка/статистика/импорт-экспорт — поверх уже готовой кинематографической дизайн-системы и честно реализованного, полностью интегрированного с базой данных «Колеса Судьбы», заменяющего собой Wheel of Names там, где у сервиса нет технической возможности для такой интеграции.
